### Redis를 적용하기 전후 성능 비교 (Postman / Curl)

성능 개선을 할 때는 반드시 **수치를 측정하면서 비교**해야 합니다. 단순히 '빨라진 것 같다'는 느낌이 아니라, 정확히 몇 ms에서 몇 ms로 단축되었는지 수치로 증명할 수 있어야 합니다.

#### 1. Redis 적용 전 (DB 직접 조회)

`BoardService`에서 `@Cacheable` 어노테이션을 주석 처리하고, API를 호출하여 성능을 측정했습니다.

- **동작 방식**: 모든 요청이 MySQL 데이터베이스까지 전달되어 `SELECT` 쿼리를 실행합니다.
- **측정 결과**:
  - 1회차: 169ms
  - 2회차: 173ms
  - 3회차: 166ms
  - **평균 속도**: **약 169ms**

#### 2. Redis 적용 후 (Cache Aside 전략)

`@Cacheable` 어노테이션의 주석을 해제하고, 첫 번째 요청(Cache Miss) 이후의 캐시 히트(Cache Hit) 성능을 측정했습니다.

- **동작 방식**: 
  - 첫 요청 시 DB 조회 후 결과를 Redis에 저장 (Cache Miss)
  - 이후 요청은 DB까지 가지 않고 Redis에서 즉시 반환 (Cache Hit)
- **측정 결과 (Cache Hit 기준)**:
  - 1회차: 36ms (초기 로딩 포함)
  - 2회차: 8ms
  - 3회차: 6ms
  - 4회차: 5ms
  - **평균 속도**: **약 10ms 내외**

#### 3. 결과 분석

| 구분 | Redis 적용 전 (DB) | Redis 적용 후 (Cache) | 개선 효과 |
| :--- | :--- | :--- | :--- |
| **평균 응답 속도** | **약 169ms** | **약 10ms** | **약 16배 이상 향상** |

- **데이터베이스 부하 감소**: 매번 실행되던 `SELECT` 쿼리와 `COUNT` 쿼리가 캐시 유효 기간(1분) 동안은 실행되지 않습니다.
- **사용자 경험 개선**: 응답 속도가 획기적으로 빨라져 더 쾌적한 서비스 제공이 가능합니다.
- **확장성**: 사용자 요청이 많아질수록 DB 부하는 기하급수적으로 늘어나지만, Redis 캐시를 통해 이를 효과적으로 방어할 수 있습니다.

> **참고 (Troubleshooting)**: 
> `Board` 엔티티의 `LocalDateTime` 필드를 JSON으로 직렬화할 때 오류가 발생할 수 있습니다. 이를 해결하기 위해 `RedisCacheConfig`에서 `ObjectMapper`에 `JavaTimeModule`을 등록하고 `GenericJackson2JsonRedisSerializer`를 사용하도록 설정했습니다.
